## ðŸ—ï¸ Database Schema Notes

### Flat vs Normalized Products Table
The system uses a **flat products table** design where:
- `category`, `brand`, `potency`, `form` are stored as TEXT (denormalized)
- This matches the Go handler expectations
- Master tables (categories, brands, etc.) exist for dropdown population
- Products reference master data by name/code, not UUID foreign keys

### Why This Design?
- **Faster queries** - No complex joins needed for product lists
- **Simpler API** - Product handler can read directly without joins
- **Historical accuracy** - Product retains original category/brand name even if master changes
- **Import/Export friendly** - CSV imports work seamlessly

```sql

CREATE TABLE IF NOT EXISTS subcategories (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  category_id uuid REFERENCES categories(id),
  name text NOT NULL,
  code text UNIQUE NOT NULL,
  is_active boolean DEFAULT true,
  created_at timestamptz DEFAULT now()
);

-- Then insert subcategories
```
